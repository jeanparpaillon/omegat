Supported file types

OmegaT supports the following source document file types for translation: plain text; HTML; XHTML; OpenDocument; OpenOffice.org; and Java resource bundles.

You can choose to have OmegaT ignore some file types by unckecking the corresponding checkboxes in the Options > File Filters dialog.

File formats are automatically detected based upon their filename extensions. Current supported file extensions are:

Plain text:
.txt for files in the OS default encoding
.txt1 for files in Latin-1
.txt2 for files in Latin-2
.utf8 for files in UTF-8
(X)HTML:
.htm
.html
.xhtml
OpenOffice:
.sx? for OpenOffice.org 1.1.5 files and earlier versions
.st? for OpenOffice.org 1.1.5 templates and earlier versions
.od? for OpenOffice.org 2.0 files and more recent versions
.ot? for OpenOffice.org 2.0 templates and more recent versions
OpenOffice.org files come in a variety of types. Word processor files have the .sxw extension for the old format or the .odt for the newer one which is the reason why we have put a "?" instead of the last letter.
Java localization ressource bundles
.properties

Files with other extensions can also be imported if their extensions are mapped to default file types. See File Filters for details on file types and extensions.

OmegaT unsupported files are copied unmodified to /target when a project is compiled. This enables, for instance, the inclusion of files for a complete web site in /source to be mirrored in /target, together with any image files (e.g. GIF, JPEG, or PNG images).

File types and encoding

The above four file types have encodings that OmegaT needs to know to be able to properly parse and display the file.

Plain text:
.txt are considered to be encoded in the OS default encoding
.txt1 are considered to be encoded in Latin-1
.txt2 are considered to be encoded in Latin-2
.utf8 are considered to be encoded in UTF-8
(X)HTML:
(X)HTML files are supposed to include encoding information. OmegaT will look for this information and parse the file accordingly. If the information is not present, OmegaT will consider the files to be in the OS default encoding.
OpenOffice:
OpenOffice.org files are encoded in UTF-8 and OmegaT parses them accordingly.
Java localization ressource bundles
Java localization ressource bundles are encoded in Latin-1 and OmegaT parses them accordingly. All the characters that are not included into ascii will automatically be converted to the proper bundle format so that you don't need to worry about anything else but getting your translation right.

Encoding settings for plain text and (x)html files can be modified from the Options > File Filters dialog. You can easily set .txt files to be considered as UTF-8 files in OmegaT by editing the Text files field of the dialog: change Source File Encoding and Target File Encoding from auto to UTF-8 for example. See File filters for more information.

Files in a supported type, with a supported file extension are expected to be encoded in the encoding set in File Filters. Files that are encoded in a different encoding may not be properly displayed since OmegaT will interpret their contents using non compatible encoding rules.

Be especially careful with plain text files. Plain text files with the .txt extension are not expected to be encoded in UTF-8.

Other files

OmegaT will look for translatable strings exclusively in files that conform to the file types extension rules above.

If you put files that are not supported by OmegaT in source/, OmegaT will not look for translatable strings in them. The files will not be modified and when the target documents are set to be created, OmegaT will simply copy them to an equivalent position in the target/ directory.

You can take advantage of this feature to translate full web sites including image files for example.

Microsoft Office Files

OpenOffice.org and related office suites (StarOffice, NeoOffice etc) possess excellent import and export filters for file format conversion–this includes filters for MS Word, Excel, and Powerpoint. Therefore, files in different formats can readily be converted to an equivalent OpenOffice.org file format, then translated in OmegaT, and finally converted back to the original format again.

Such round-trip conversion rarely generates problems but you may want to make tests on a few samples to confirm that your files can go through this process without major breakdowns.
