Translation Memory and Glossary Files

  1. Translation memory files
  2. Glossary Files

Translation memory files created by OmegaT are standard TMX 1.1 files.
For more information on the TMX specification, see http://www.lisa.org/standards/tmx/specification.html.

Note! OmegaT 1.6 is a transition version toward full TMX 1.4b compliance in OmegaT 2.0.

Translation memory files

OmegaT generated files

OmegaT generates translation memory files that use only a subset of the TMX specification.

TMX files generated by OmegaT for a specific project are described in Translation project files.

A standard OmegaT TMX file looks like the following:

When you want to work with TMX files created by older versions of OmegaT make sure you take the segmenting into account: either you work in paragraph segmenting in OmegaT 1.6 or you manually (or with the use of a conversion tool) convert paragraph segmented translation units (<tu>) into sentence segmented translation units.

Inline formatting information support

As described in OmegaT tags, OmegaT takes formatting information into account when translating. OmegaT tags are specific to OmegaT since they only refer to bits of information kept in the internals of OmegaT, they only point at the bits, they don't display them.

While this behavior is amply sufficient for projects that interact with OmegaT translation memories exclusively, it impairs OmegaT's ability to export tmx files that are fully understood by other translation memory tools.

To use OmegaT 1.6 tmx files in other tools, it is advised to remove the OmegaT specific tags. See Textual information support for simple conversion technics.

Lets consider the following segment from a HTML file:

<b><a href="http://www.omegat.org/EN/home.html">English</a> | <a href="http://www.omegat.org/ES/home.html">Spanish</a></b>

In OmegaT it is displayed the following way:

<segment nnnn><a0>English</a0> | <a1>Spanish</a1><end segment>

<b> and </b> are not taken into account because only inline formatting that affects parts of the segment is considered by OmegaT. The bold tag pair affects the whole segment and thus is not considered as inline information in OmegaT 1.6.

The corresponding OmegaT tmx translation unit would look like this:

 <tu>
  <tuv lang="EN-US">
    <seg>&lt;a0&gt;English&lt;/a0&gt; | &lt;a1&gt;Spanish&lt;/a1&gt;</seg>
  </tuv>
  <tuv lang="JA-JA">
    <seg>&lt;a0&gt;英語&lt;/a0&gt; | &lt;a1&gt;スペイン語&lt;/a1&gt;</seg>
  </tuv>
 </tu>

This marking is specific to OmegaT but allows it to reproduce the totality of the formatting in the target segment. The translated html segment will reproduce the structure of the original html segment:

<b><a href="http://www.omegat.org/EN/home.html">英語</a> | <a href="http://www.omegat.org/ES/home.html">スペイン語</a></b>

Internally, OmegaT 1.6 supports inline formatting, but is not able to produce TMX level 2 compliant translation units for use in other TMX level 2 compliant tools.

The above translation unit would look like the following if OmegaT were TMX level 2 compliant:

 <tu>
  <tuv lang="EN-US">
    <seg><bpt><b></bpt>
    　　　<bpt><a href="http://www.omegat.org/EN/home.html"></bpt>English<ept></a></ept> |　
    　　　<bpt><a href="http://www.omegat.org/ES/home.html"></bpt>Spanish<ept></a></ept>
    　　　<ept></b></ept></seg>
  </tuv>
  <tuv lang="JA-JA">
    <seg><bpt><b></bpt>
    　　　<bpt><a href="http://www.omegat.org/EN/home.html"></bpt>英語<ept></a></ept> |　
    　　　<bpt><a href="http://www.omegat.org/ES/home.html"></bpt>スペイン語<ept></a></ept><ept>
    　　　</b></ept></seg>
  </tuv>
 </tu>

Where bpt is a tag pair that indicates where is the inline formatting information beginning tag (bpt means begin paired tag), and ept is a tag pair that indicates where is the inline formatting information ending tag (ept means end paired tag).

Textual information support

It is possible to remove all the inline formatting information to keep exlusively the textual information (TMX support at level 1).

In that case, removing the extra OmegaT tags with a simple search/replace operation using regular expressions would produce a satisfying tmx file.

Using the regular expressions described in Regular Expressions in a text editor, we could search for &lt;/?[a-z]+\d+&gt; and replace by nothing to produce the following OmegaT tmx file:

 <tu>
  <tuv lang="EN-US">
    <seg>English | Spanish</seg>
  </tuv>
  <tuv lang="JA-JA">
    <seg>英語 | スペイン語</seg>
  </tuv>
 </tu>

With modern text editors, such an operation on multiple files at once is trivial and will allow you to export OmegaT tmx files so that they can be easily used by other translation memory tools.

See the TMX specification for more information on the tmx file format and on tmx compliance.

Importing translation memories

OmegaT 1.6 is able to use any translation memory that conforms to the TMX standard.

It does so by ignoring the parts that it cannot use, specifically TMX tags attributes and TMX inline tags, as well as tmx tags such as <prop> and <note>.

The following tmx files are parsed by OmegaT, but the sections in bold characters will not be meaningful to OmegaT. In the case of the Trados file, the contents of the <seg> pair will be displayed in the match window with all the inline characters.

Glossary Files

Currently, OmegaT does not provide for glossary terms operations. The glossary window indicates glossary items that are present in the current segment. Glossary operations will be added to newer versions of OmegaT

Glossary files generation

Glossary files are plain text files. These files contain rows (lines) of three tab-delimited columns with source and target terms in the first and second columns, respectively. The third column in each line can be used for additional information (e.g. comments, descriptions, references, etc.). The encoding used in these files is either the system default encoding (with a .tab filename extension) or UTF-8 (with a .utf8 filename extension).

Users with access to glossaries created by Trados Multiterm's text export function can use them directly without modification, provided they are given the required .tab file extension and the source and target term fields are the first two fields as required.
