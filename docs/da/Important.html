Precautions against data loss!

OmegaT is a robust application. It is not prone to crashes or data loss and corruption. However, you should take precautions against data loss when using OmegaT, just as with any other application.

In general:

  • use OmegaT on short texts until you are familiar with it.
  • read Translation: Save, Revision, and Compilation carefully; before working on any important projects.
  • to avoid any loss of data, manually save a project's translation memory (Project > Save Translation Memory), even though OmegaT automatically saves translation memory at very short time intervals.
  • Make regular backup copies of the file /omegat/project_save.tmx. A good idea would be to add a serial number or the date to each copy, for instance, /omegat/project_save.tmx-01. or /omegat/project_save.tmx-07-06-05. Backups can be transferred to other media such as floppy disk, CD-RW, or other hard disk if they are available. Then project_save.tmx can be conveniently restored from backup if something goes wrong.
  • Make compilations at regular intervals and check that the compiled file contains the latest version of a translation. In particular, this should be done when first starting out with OmegaT.
  • Take particular care when making any changes to the files in /source in the middle of a project. Sometimes it might be necessary to make changes to source files, for example to remove hard line breaks from the middle of sentences. However, if the source file is modified after translation has begun, OmegaT may be unable to find a segment that has already been translated. Similarly, changes to the sentence segmentation rules while working on a project may deeply affect your translation. Always backup the source file itself, or the whole project, before making any changes to the source file(s) or to the sentence segmentation rules.

Please keep in mind that OmegaT creates a backup of a project's translation memory in the project_save.tmx.bak file in the project's /omegat subdirectory whenever a project is opened and before any modifications are made.

Most users will be able to get started with the aid of this manual. If problems arise then post a message to the OmegaT users group located at http://groups.yahoo.com/group/omegat with details of the particular situation. Do not hesitate to post in the language you feel the most familiar with.

Error Logging

OmegaT creates a log file (log.txt) in the application's install directory with errors encountered during operation. If errors are encountered they can be posted to the OmegaT users group or, if they are definitely programming bugs, submitted to the OmegaT sourceforge Bugs page (accesible via http://sourceforge.net/projects/omegat.)

An OmegaT error log looks like this:

===================================================================
OmegaT-1.6.0.RC2 (Sun Nov 06 23:33:44 JST 2005)  Locale fr_FR

ERROR Search thread died unexpectedly! Please close search window.
java.lang.NullPointerException
	at java.lang.reflect.Array.getLength(Native Method)
	at org.omegat.util.StaticUtils.buildFileList(StaticUtils.java:104)
	at org.omegat.util.StaticUtils.buildFileList(StaticUtils.java:119)
	at org.omegat.core.threads.SearchThread.searchFiles(SearchThread.java:309)
	at org.omegat.core.threads.SearchThread.run(SearchThread.java:144)
----------------------------

Bug reports

Before reporting a bug, make sure that you have thouroughly checked the documentation. What you see may instead be a characteristic of OmegaT that you have just discovered. If you check the OmegaT log and you see words like "ERROR" or "died unexpectedly" (see above) then you are on to something.

The next thing you do is confirm what you found with other users, to make sure this has not already been reported. You can verify the bug report page at SourceForge too. Only when you are sure you are the first to have found some reproductible sequence of event that triggered something not supposed to happen should you file a bug report:

It's pretty easy to remember the rule for a good bug report.
Every good bug report needs exactly three things.

  1. Steps to reproduce,
  2. What you expected to see, and
  3. What you saw instead.

You can add copies of files, portions of the log, screenshots, anything that you think will help the developers with finding and fixing your bug.

To browse the archives of the user group, go to:
http://groups.yahoo.com/group/OmegaT/

To browse the bug report page and file a new bug report if necessary, go to:
http://sourceforge.net/tracker/?group_id=68187&atid=520347
To keep track of what is happening to your bug report you may want to register as a Source Forge user.
