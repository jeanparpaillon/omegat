Tagged (Formatted) Files Text Editing

  1. OmegaT tags
  2. Tag operations
  3. Tag Validation
  4. Java localisation bundles

Plain text files only require typing the correct translation in the editing field. Other file types that OmegaT supports natively sometimes require specific knowledge of how OmegaT handles them. This chapter describes how OmegaT handles (X)HTML files, OpenOffice.org files and Java localisation bundles.

OmegaT tags

Formatting information present in the source file usually needs to be reproduced in the target file. OmegaT displays the in-line formatting information of supported formats (at present HTML, XHTML, OpenDocument, and OpenOffice.org) as tags. Tags are usually not taken into account when considering string similarity for matching purposes. Tags that you have reproduced in the translated segment will be present in the translated document.

Examples in this chapter are taken from this user manual as displayed in OmegaT. Some files have purposedly been converted to OpenOffice.org format to ensure completeness of the examples. We use the notation HTML tags to refer to HTML and XHTML tags, and OOo tags to refer to OpenDocument, and OpenOffice.org tags.

Pairs and singletons

Tags come either in singletons or in pairs.

  • Single tags indicate formatting information that does not affect the surrounding text (extra space or line break for example).
    Single tags:
    <segment 2132><b0><Ctr+N></b0>, <br1><b2><Enter></b2>, <br3><b4><Tab></b4><s5>*</s5><end segment>
    <br1> and <br3> are single tags and do not affect any surrounding text.
  • Paired tags usually indicate style information that applies to the text between the opening tag and the closing tag of a pair.
    Paired tags:
    <segment 3167>Log file (<f1>log.txt</f2>) for tracking operations and errors.<end segment>
    <f1> and </f2> are paired and affect log.txt

Tags numbering

Tags in OmegaT are numbered. Numbering depends on the source file format.

HTML tags:
<segment 2132><b0><Ctr+N></b0>, <br1><b2><Enter></b2>, <br3><b4><Tab></b4><s5>*</s5><end segment>
Tags in HTML source files are numbered in an incremental way by tag group. The first group (pair or singleton) gets the number 0, the second the number 1 etc.
Here, the (pair) group <b0> </b0> gets number 0 and the (singleton) group <br1> gets number 1.
OOo tags:
<segment 3167>Log file (<f1>log.txt</f2>) for tracking operations and errors.<end segment>
Tags in OOo source files are numbered in an incremental way by tag. The first tag (tag in a pair or singleton) gets the number 1, the second the number 2 etc.
Here, the tag <f1>, opening a pair, gets number 1 and the tag </f2>, closing the pair, gets number 2.

The different numbering pattern helps you remember that in HTML files, you have to focus on tag pairing, while in OOo files, tag order is more important.

Tags and sentence segmenting

OmegaT creates its tags before sentences segmenting is applied. Depending on the segmenting rules, tags may look like they do not respect the above rules of pairing and numbering.

Let's consider the following segment without sentence segmenting:

<segment 1554> <c0>Before: \. After: \s</c0><br1> <b2>Note:</b2> "\." means the character "."<br3> "\s" means any white space character <end segment>

There seems to be no problem here. Now, if we apply the default segmenting rules on this segment (the default OmegaT behavior) we will have the following result:

<segment 1990> <c0>Before: \. <end segment>
<segment 1991> After: \s</c0><br1> <end segment>
<segment 1992> <b2>Note:</b2> "\." means the character "."<br3> <end segment>
<segment 1993> "\s" means any white space character <end segment>

Here, we see that segments taken one by one do not respect the pairing and numbering rules. In some cases that may even cause problems in the translation, when tags must be displaced in the target language to reflect the word order of that language.

OOo Tags and segment boundaries

Only OOo tags are concerned by this item. HTML tags are not.

Tags can sometimes fall on segment boundaries (ie. just after <segment nnnn> or just before <end segment>). OmegaT usually does not display OOo tags that fall on segment boundaries.

When the target language structure requires word order to be changed, it may sometimes be difficult to displace formatting information so that it sticks to the formatted string since one of the tag is missing. This issue is well known and will be fixed in a future version of OmegaT.

If non displayed tags must be displaced insert a mark in the target segment and correct the formatting in the target document with a compatible application.

Tags falling on segment boundaries (current, faulty, behavior):
<segment 3134>Instant Start tutorial</a1> <f2>new!<end segment>

Proper support for tags falling on segment boundaries should have OmegaT display the above segment the following way:

Tags falling on segment boundaries (future, correct, behavior):
<segment 3134><a1>Instant Start tutorial</a2> <f3>new!</f4><end segment>

Tag operations

Care must be exercised with tags. If they are accidentally changed, the formatting of the final file may be corrupted. In the worst case scenario a file may not open altogether; this occurs with OpenOffice.org files in particular.

Although custom formatting cannot be performed from within OmegaT, it is possible to apply some degree of control over the existing format.

OOo tags manipulation

OpenDocument and OpenOffice.org tags must be preserved as is. These tags cannot be deleted or have their sequence changed. However, it is possible to:

  • move tags to a different part of the segment; as long as they remain in the same order;
  • cancel tags by placing them next to each other.

For instance, if the tagged sentence

<segment 3270> Log file (<f1>log.txt</f2>) for tracking operations and errors. <end segment>

is changed to

<segment 3270> Log file (<f1></f2>log.txt) for tracking operations and errors. <end segment>

the formatting of log.txt is effectively removed.

HTML tags manipulation

In HTML or XHTML the following rules for tags are applicable:

Duplication is permitted e.g.:
<segment 3116> Help menu includes access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>

to duplicate tags, just copy them in the position of your choice (remember that overlaping of paired tags is never allowed, see Overlap below):

<segment 3116> Help menu <b0>includes</b0> access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>
Deletion is permitted e.g.:
<segment 3116> Help menu includes access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>

to remove formatting you don't need, just delete the related tag pairs:

<segment 3116> Help menu includes access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>
Change of order is permitted:
<segment 3116> Help menu includes access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>

can be changed to:

<segment 3116> Help menu includes access to a new <b1>About</b1> dialog as well as the <b0>Users Manual</b0>. <end segment>
Nesting is permitted e.g.
<segment 3116> Help menu includes access to a new <b0>About</b0> dialog as well as the <b1>Users Manual</b1>. <end segment>

can be changed to

<segment 3116> Help menu includes access to a new <b0>About dialog as well as the <b1>Users Manual</b1>.</b0> <end segment>
Overlap is not permitted. Tagging like this:
<segment 3116> Help menu includes access to a new <b1>About dialog as well as the <b0>Users</b1> Manual</b0>. <end segment>

is not allowed and will corrupt the file. Insertion of an opening tag followed by a second opening tag requires the closing tag of the second opening tag to come before the closing tag of the first.

Tag Validation

The validate tags function ( Tools > Validate Tags) detects changed tags (whether done deliberately or by accident), and indicates the affected segments. Use of this function will open a dialog with any suspected broken or bad tags in a document.

This function can be useful for tracking down bugs in a translated tagged text. This is often a problem in OpenDocument or OpenOffice.org files that will not open due to tag problems created in the process of translation. Fixing the tags and recreating the target documents again can often remedy the problem.

Java localisation bundles

Java localisation bundles are not to be considered as tagged files. Still, their translation is not as straight forward as plain text files so we include here a reasonably detailed section about their specificities. Most examples are taken from the OmegaT GUI bundle.

Warning! Special characters are used in bundles to indicate user interface access accelerators, variables, escaped characters (end of line etc.), you must reproduce these characters in the relevant position of your translation.

Translation context

Although the bundle internal structure is relatively complex, OmegaT hides the complexity and only gives you access to the string to translate. If you are not confident about the context of the string you are translating, open the file in a text editor and consult it if necessary, while translating in OmegaT.

A few lines of the OmegaT bundle:

# The version number is taken from Version.properties, key 'version'
# and inserted into the following template
version-template=OmegaT-{0}
# buttons
BUTTON_OK=&OK
BUTTON_CANCEL=&Cancel

OmegaT will only display the part after the "=" equal sign:

OmegaT-{0}
<segment 4345> OmegaT-{0} <end segment>

&OK
<segment 4346> &Ok <end segment>

Comments

Comments do not appear in OmegaT, but you may want to read them while translating (see Translation context above).

Here are two examples of comments:

# MainWindow TF_INTRO_MESSAGE=OmegaT Translation Memory CAT Tool

# NOTE: segment start is assumed to end with "0000" string to overwrite
# with entry number. If zeros not detected, entry number will not be
# displayed TF_CUR_SEGMENT_START=<segment 0000>

Any line in the .properties file that begins with a hash sign (#) (also known as a pound sign in the United States), is a comment. These comments are for the benefit of the translator and the programmer, and need not be translated. None of the comments will appear anywhere in the user interface, so there's no need to translate them.

Accelerators

The acccelerator enables the user to navigate the menus using his keyboard. Accelerators are letters inside a function name directly accessible from the keyboard that trigger that function. In Java bundles they are indicated with "&". Such keyboard access is not conformant with MacOSX GUI guidelines but is common on Windows or Linux.

Here are two examples with accelerators:

BUTTON_SEARCH=&Search
the accelerator for "Search" is set to letter "s".

BUTTON_TO_DEFAULTS=De&faults
the accelerator for "Defaults" is set to letter "f".

Try to use different letters for the accelerator in successive strings, because these strings often appear together on a menu or dialog. Try to use the same letter for the accelerator as is used in English.

Examples:

In Afrikaans, the translation for "Search" and "Defaults" is "Deursoek" and "Verstek".

Incorrect:
* BUTTON_SEARCH=Deur&Soek (no need to capitalize "s").
* BUTTON_SEARCH=Deur &soek (no need to insert a space).
* BUTTON_SEARCH=Deursoek& ("&" is not set to any letter).
* BUTTON_SEARCH=& Deursoek ("&" is not set to any letter).

Correct:

* BUTTON_SEARCH=Deur&soek
* BUTTON_SEARCH=&Deursoek

Incorrect:

* BUTTON_TO_DEFAULTS=Ver&Stek (no need to capitalize "s").
* BUTTON_TO_DEFAULTS=&fVerstek (adding "f" makes the translation incorrect).
* BUTTON_TO_DEFAULTS=Verstek&f (adding "f" makes the translation incorrect).

Correct:

* BUTTON_TO_DEFAULTS=&Verstek
* BUTTON_TO_DEFAULTS=Ver&stek

Variables

Here are three example of variables:

HF_ERROR_EXTLINK_TITLE=Cannot open the link {0}
CT_FUZZY_X_OF_Y=Analyzing strings - {0} of {1} completed
SOME_FICTICIOUS_EXAMPLE=Project saved on {0} at {1} by {2}

The {0}, {1} and {2} are placeholders for variables. When OmegaT displays this string to the user, it automatically adds numbers or pieces of text instead of the {0}, the {1} or the {2}.

Examples:

In Afrikaans, the strings are usually translated as follows (notice the last example, in which the order of the variables are different owing to the different grammatical structure of Afrikaans):

HF_ERROR_EXTLINK_TITLE=Cannot open the link {0}
produces the following string: Cannot open the link foo.bar
which should be translated: Kon nie die skakel foo.bar open nie

The OmegaT segment should thus be:
HF_ERROR_EXTLINK_TITLE=Kon nie die skakel {0} open nie

Similarly:

CT_FUZZY_X_OF_Y=Analyzing strings - {0} of {1} completed
produces the following string: Analyzing stings - 23 of 100 completed
which should be translated: Ontleed tans stringe - 23 van 100 afgehandel

The OmegaT segment should thus be:
CT_FUZZY_X_OF_Y=Ontleed tans stringe - {0} van {1} afgehandel

And:

SOME_FICTICIOUS_EXAMPLE=Project saved on {0} at {1} by {2}
produces the following string: Project saved on 2005-11-23 at 23:55 by John Doe
which should be translated: Projek om 23:55 op 2005-11-23 deur John Doe gestoor

The OmegaT segment should thus be:
SOME_FICTICIOUS_EXAMPLE=Projek om {1} op {0} deur {2} gestoor

Escaped characters

Some letters or signs used in ordinary, human writing have special meaning to a computer, and for that reason these characters must be escaped to make OmegaT treat them correctly and display them correctly.

Some characters are escaped so that they display (for example, the colon). Some characters are escaped so that they don't display (for example, the newline-n).

Here are some examples of escaped characters (in the .properties file, characters are escaped using a backslash):

HF_CANT_FIND_HELP=Cannot locate help file\:
will be displayed on screen as:
Cannot locate help file:

RBFH_ERROR_ILLEGAL_U_SEQUENCE=Encountered illegal \\u sequence
will be displayed on screen as:
Encountered illegal \u sequence

SOME_FICTITIOUS_STRING=Hello\nworld!
will be displayed on screen as:
Hello
world!

HTML in strings

Most of the strings in the .properties file will be displayed in OmegaT as menu items or dialog boxes. However, a few of them will be displayed in a view box that uses HTML (the haiku error messages are one example).

Here is an example of such a string (notice also the escaped colon):

HF_HAIKU_2=Wind catches lily<p>Scatt'ring petals to the wind \:<p>Your page is not found

This string will be displayed in the view box as:

Wind catches lily
Scatt'ring petals to the wind:
Your page is not found

Dealing with newlines

To display multiple lines in the OmegaT user interface, escaped newline-n characters are used to force a break in the line. Do not use newlines in the translation if there are no newlines in the English text, and try to keep as many lines in the translation as in the English text.

Here is an example of newlines and their effects:

SOME_FAKE_STRING=This\nis\nmy\nlife

This will be displayed as:

This
is
my
life

Notice that there are no spaces before or after the newline-n characters. If you insert spaces, OmegaT will display spaces.

Here is an example with incorrectly inserted spaces:

SOME_FAKE_STRING=This\n is\n my\n life

This will be displayed as:

This
 is
 my
 life

Multiline strings

For reasons of legibility, the programmer may have broken a long string into several lines in the .properties file. The line breaks in the .properties file will not affect the way the string is displayed on screen.

Here are two examples of multiline strings:

YET_ANOTHER_FAKE_STRING=This is an example of\
how one would write a multiline string\
and how one would write it.

will be displayed: This is an example of\how one would write a multiline string\and how one would write it.

STILL_ANOTHER_FAKE_STRING=\
 This is another example of how one would write\
 a multiline string and how one would write it.

will be displayed: STILL_ANOTHER_FAKE_STRING=\ This is another example of how one would write\ a multiline string and how one would write it.

The multiline string ends with a backslash, and may or may not be indented.

Some multiline strings end on both a newline-n and a backslash. Those strings will indeed be displayed as multiple lines in the OmegaT user interface, because of the newline-n characters.
