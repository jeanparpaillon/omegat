Plain Text in OmegaT

Plain text files contain exclusively textual information. Plain text files have no clearly defined way to inform the computer which language they contain. Very roughly speaking, that means that the computer considers by default that the file contents is in the same language as the computer itself.

A plain text file is in most of the cases a file with a name ending in .txt.

Garbled displays?

If you are Russian, it is very likely that your computer works in Russian too: the menus are in Russian, the files you open will be in Russian etc. In most cases, the computer makes the right assumption regarding the contents of files in general: they all contain Russian and nothing Russian characters could not display.

Now, if you are a Russian translator who translates from Japanese, the Japanese files you will get, if they are plain text files will most probably be considered by the computer to be files containing Russian. Because there is no information in the file itself that indicates to the computer in which language they are written.

The Japanese file contents could be: 

   OmegaTとは、コンピュータを利用した翻訳ツールです。

But your text editor could very well display it like this: 

   OmegaTВ∆ВЌБAГRГУГsГЕБ[Г^ВрЧШЧpВµВљЦ|ЦуГcБ[ГЛВ≈ВЈБB

Because it expects the contents to be Russian... But this is not Russian. This is Japanese characters wrongly displayed as Russian characters.

OmegaT is no different. OmegaT considers that plain text files contain text that can automatically displayed by using the computer's defaults. That works well when the computer works in French and if you get English files, or when the computer is German and if you get Italian files.

Character sets and encodings

Why would that work with English and French but not with Russian and Japanese? Because English and French share a common character set. Namely Latin-1, or a variation.

Until recently, Russian and Japanese did not share any character sets. Most current Russian characters sets do not cover Japanese and reciprocally. The result is as shown above.

The Japanese client works with a Japanese computer and creates text files that contain Japanese. The character set selected by the client computer will depend on the operating system and of other settings, but it is very unlikely that the chosen (Japanese) character set will be correctly interpreted by the Russian computer.

Now, how the textual information in the specified character set is physically transmitted (i.e. how is it written in the file for the computer to interpret and display) depends on an encoding. When the computer reads the file, it "decodes" the information according to the encoding and displays it according to the character set. Roughly speaking, one encoding corresponds to one character set...

The OmegaT solution

There are basically 3 ways to fix this in OmegaT.

  1. The drastic way:
    • Tell OmegaT that your plain text files ending in .txt should be interpreted as the encoding you specify.
  2. The selective way:
    • Change your plain text "foreign" file names to something you like (something.japanese for example);
    • Tell OmegaT that files ending with ".japanese" should be interpreted as the encoding you specify.
  3. The "Unicode" way:
    • Open your "foreign" file in a text editor that correctly interprets its encoding (or proceed with trial and error);
    • Save the file in the "UTF-8" encoding;
    • Change its name from something.txt to something.utf8. OmegaT will interpret the file as a UTF-8 file.

Currently, OmegaT is set to understand plain text files the following way: (you can check that yourself by selecting the item File Filters in the menu Options. Click on the Text Files row and then on the Edit button.)

  • .txt files are automatically (auto) intepreted by OmegaT as being ecoded in the computer's default encoding.
  • .txt1 files are intepreted by OmegaT as being ecoded in ISO-8859-1 (an encoding that covers most Western Europe languages).
  • .txt2 files are intepreted by OmegaT as being ecoded in ISO-8859-2 (an encoding that covers most Central and Eastern Europe languages).
  • .utf8 files are intepreted by OmegaT as being ecoded in UTF-8 (an encoding that covers almost all languages in the world).

Of course, the .txt1, .txt2, and .utf8 file endings do not exist in real life. OmegaT has just prepared for you a short list that will make it easier for you to deal with some foreign files.

So, now, when you have a French file (in_french.txt for example) to translate on a Japanese computer, since it is very likely to have been created as a ISO-8859-1 encoded file, you know that by just adding a 1 at the end of its name (in_french.txt1) OmegaT will properly interpret its contents as being in ISO-8859-1... Et voilà!
